\section{Introduction}

Machine learning development is iterative and exploratory.
Practitioners assemble data from heterogeneous sources,
clean and preprocess it,
build candidate models,
evaluate them,
revise feature engineering choices,
and repeat the cycle until the model
satisfies the target criteria~\cite{haakman2021ai,amershi2019software,sculley2015hidden}.
Each cycle is driven by feedback,
closely resembling prominent development processes,
such as Agile~\cite{betz2018managing} and CRISP-DM~\cite{martinez-plumed2021crisp-dm},
that advocate for short feedback loops
and frequent revision in response to what each cycle reveals.

Jupyter notebooks have become the dominent medium for developing ML prototypes
since they support a feedback-driven workflow
through a read-eval-print loop style of development.
Practitioners write a small unit of code within a \emph{cell},
execute it,
read the result,
and decide what to do next~\cite{pimentel2019large-scale,quaranta2021kgtorrent,psallidas2019data,perkel2018why}.
The output of each cell is a unit of feedback,
that is made visible by the notebook structure.

We define a \emph{feedback statement}
as a statement that the practitioner intentionally adds to the code
to obtain information about the program's execution.
This includes
\textbf{(1)} exploratory statements (EXP)
that display values for the practitioner to interpret
and \textbf{(2)} validation statements (VAL)
that enforce a condition programmatically.
The former kind of feedback,
is obtained from the last statement in a code cell,
whose value is displayed beneath the cell when it executes.
The practitioner reads the output
and judges whether the program is behaving correctly.
This is the primary mode of exploratory programming,
that prior studies identify
as central to data science work
in Jupyter notebooks~\cite{kery2018story,rule2018exploration}.
In the latter,
the condition is encoded in the statement,
and a violation halts execution with an explicit signal.
This form of feedback
is obtained from \lstinline{assert} statements,
used by practitioners to validate assumptions regarding a piece of code.
Pimentel et al.~\cite{pimentel2019large-scale} report
that unit tests are virtually absent from Jupyter notebooks.
We therefore study assertions directly
rather than the unit test functions that contain them.
% TODO: is this clarification necessary?
Note that
both forms of feedback are intentional.
A bug that produces an error message also provides feedback,
but does not satisfy our definition
because the practitioner did not author the bug to obtain feedback.

Feedback statements encode the practitioner's mental model
of what the code should be doing
and what could go wrong.
For example,
a statement that displays the shape of a tensor after a transformation
encodes the expectation that the transformation should preserve some dimension.
Similarly,
A statement that asserts the absence of missing values before feature selection
encodes a precondition for the next step.
Existing empirical work on ML notebooks
has examined points where the execution halts
by analyzing crashes~\cite{wang2025why} and bugs~\cite{desantana2024bug}
that occur within Jupyter notebooks.
In contrast,
this study explores
what practitioners check
when execution does not halt.
We do this to complement prior crash-based studies,
which cannot observe these statements
because they exist to prevent crashes
or to confirm that no crash has occurred.

We mine 297,851 publicly available Python Jupyter notebooks
from GitHub and Kaggle,
and extract 1,092,780 feedback statements.
We sample 815 statements
using proportional stratified sampling
from semantic clusters obtained from CodeBERT embeddings.
We use grounded theory and open coding
to code each statement along two dimensions:
\textbf{(1)} the \emph{intent} of the statement,
which captures what the practitioner is trying to learn or confirm,
and \textbf{(2)} the \emph{pipeline stage},
which captures where in the ML workflow the statement appears.

We use a combination of statistical hypothesis testing
and qualitative analysis
to answer the following research questions.

\begin{description}
  \item[RQ1.] \textbf{What feedback mechanisms do practitioners use in ML notebooks,
    and how do they vary by platform?}
  \item[RQ2.] \textbf{What does the practitioner intend to learn or confirm
    from each feedback statement?}
  \item[RQ3.] \textbf{Where in the ML pipeline do practitioners author feedback statements?}
  \item[RQ4.] \textbf{How does the feedback taxonomy relate to known crash modes in ML notebooks?}
\end{description}

This study makes four contributions.
\textbf{First},
we provide an empirical characterisation of feedback practice in ML Jupyter notebooks,
and highlight the platform-level differences between GitHub and Kaggle.
\textbf{Second},
we present a taxonomy of feedback statements in ML Jupyter notebooks.
The taxonomy is created using grounded theory,
and is coded along the intent and pipeline stage.
\textbf{Third},
we establish a mapping between our feedback taxonomy
and the crash taxonomy of Wang et al.~\cite{wang2025why}.
The mapping surfaces the duality
between defensive practice and observed failure,
and identifies regions of the failure space
that crash analysis alone cannot capture.
\textbf{Finally},
we release a publicly available dataset
consisting of 1M+ feedback statements,
suitable for replication and downstream tooling research.

We find that
feedback in ML notebooks is overwhelmingly exploratory,
and account for 85\% of the corpus.
Formal assertions account for the remaining 15\%
and are concentrated almost entirely in GitHub.
Additionally, we find that
the two platforms host qualitatively different modes of ML work.
Kaggle notebooks concentrate 58\% of feedback in the data preparation stage
and contain almost no model construction code.
GitHub notebooks on the other hand,
distribute feedback across data ingestion, model construction, training, and evaluation.
Finally, both our study and the Wang et al.\ crash study
identify data preparation as the empirical centre of gravity in ML notebook development.
Wang et al.\ report that 33\% of crashes occur in data preparation,
while we report that 40\% of feedback statements occur in the same stage.
The two studies with inverted lenses
together indicate where practitioner attention concentrates.

As ML pipelines developed within Jupyter notebooks become business-critical,
they are frequently transitioned into automated scripts
% TODO: find better citations to support this
for production environments~\cite{kery2018story,rule2018exploration}.
Our findings suggest
a crutial need to shift from implicit feedback through exploratory statements,
to explicit validation performed using assertions.
This asymmetry motivates the development of validation tooling,
that integrates with the current iterative notebook workflow.

The research goal is to \emph{characterize} the feedback statements used in ML
Jupyter notebooks. This research goal guides the data collection and analysis
methodology outlined in this section.

\subsection{Data Collection}

Figure~\ref{fig:method} illustrates the
data collection,
filtering,
sampling,
and manual analysis
employed in this study
to gather and analyze
Jupyter notebooks written in Python
from GitHub and Kaggle.

GitHub is a popular platform to host and collaborate on software projects.
It offers extensive tools for project management
such as issue tracking,
pull requests,
code reviews and tooling for continuous integration and delivery.
In contrast,
Kaggle is focused specifically on data science and machine learning.
It provides integrated computational resources,
a rich repository of datasets,
and a platform to host machine learning competitions.
While GitHub supports formal project workflows,
Kaggle encourages a more informal approach to data exploration and modeling,
making it a preferred choice
for data science practitioners
looking for hands-on experience
with developing machine learning models.

We use GitHub's advanced search syntax~\footnote{https://docs.github.com/en/search-github/searching-on-github/searching-code},
~\footnote{We use the following search query: \lstinline[language={}]$language:"Jupyter Notebook"$}
to collect \emph{all} Jupyter notebooks from public repositories
authored on June 22, 2023.
We use the pre-existing dataset KGTorrent~\cite{quaranta2021kgtorrent,kgtorrent-zenodo} for Kaggle
since it does not support advanced code-based search like GitHub.
To the best of our knowledge,
KGTorrent is the largest dataset of Python Jupyter notebooks obtained from Kaggle.
Consequently, we start with 297,851 notebooks (283 GB)
comprising of 49,090 notebooks from GitHub and 248,761 notebooks from Kaggle.

\subsection{Data Filtering}

For the notebooks collected from GitHub,
we excluded notebooks that did not contain Python code.
This step is not applied to the Kaggle notebooks
since KGTorrent only contains Python notebooks.

Since the focus of this study is to analyze Python code in Jupyter notebooks,
we remove notebooks with invalid JSON structures
and notebooks that do not contain any code cells.
To concentrate specifically on machine learning projects,
we only include notebooks that import at least one major ML library:
Scikit Learn~\cite{pedregosa2011scikit-learn},
PyTorch~\cite{paszke2017automatic},
TensorFlow~\cite{abadi2015tensorflow},
or Keras~\cite{chollet2015keras}.
While this approach may exclude some
ML projects using alternative libraries,
prior research by Psallidas et al.~\cite{psallidas2019data}
examined six million GitHub Jupyter notebooks
and confirmed that most ML-oriented notebooks utilize these frameworks.

We deliberately exclude notebooks
that import Pandas, NumPy, and SciPy.
Despite their widespread usage in ML,
these libraries serve numerous purposes beyond ML
and are not specifically designed for model training.

After applying these
filters, we exclude 102,276 (41\%) notebooks
and retain 195,575 notebooks (25,947 from GH and 169,628 from KG) for analysis.

We extract the contents of \emph{all} code cells from the notebooks
by parsing the underlying JSON structure.
We then use the \lstinline{ast} module provided by the Python standard library,
to extract \emph{all} assert, and last statements from the code cells.
The raw contents of each cell are converted into an abstract syntax tree (AST).
The \lstinline{NodeVisitor} class from the \lstinline{ast} module is used
to traverse the AST and collect all \lstinline{Assert} nodes.
To collect assertions written using Python modules not included in the standard distribution,
we also collect all function \lstinline{Call} nodes
that contain the string ``assert'' in their name.
To collect the last statements,
we first remove all cells that did not produce any output
and consequently extract the last top-level node from the AST of the code cell.
The raw Python code is regenerated from the ASTs
using the \lstinline{unparse} method provided by the \lstinline{ast} module.

We obtain a population of 220,095 statements (80,034 asserts and 140,951 lasts) from GitHub
and 872,685 statements (9,572 asserts and 863,113 lasts) from Kaggle.

\subsection{Sampling}

In a premiminary analysis,
we apply clustering to the entire dataset,
and manually analyze 100 statements (50 from GH and KG) sampled randomly from GH and KG.
we find that duplicate statements conflate the cluster structure
and cause a random sample to under-represent diverse patterns.
We therefore remove exact duplicate statements prior to clustering.
We delimerately trade-off diversity with representativeness for the taxonomy.

We use CodeBERT to obtain semantic code embeddings for each unique statement.
Semantic embeddings are appropriate
because statements with different surface forms can express the same intent.
E.g., \lstinline{assert X == Y} and \lstinline{assert X is Y}
are syntactically distinct but semantically related.
We apply UMAP~\cite{McInnes2018} to reduce the embedding dimentionality to 10.
We then apply HDBSCAN~\cite{McInnes2017} to cluster the reduced embeddings
into groups based on semantic similarity of the embeddings.

We obtain 277 clusters from GH and 1031 clusters from KG.
Prior to sampling,
we collapse small clusters to prevent excessive fragmentation of the strata.
We rank clusters by size and compute their cumulative coverage of the entire population.
Clusters that together account for the top 70\% of statements are retrained as individual strata.
All remaining clusters are merged into a single residual stratum
to ensure that each retained stratum is large enough to yield a meaningful proportional allocation.

We apply proportional stratified sampling
with a 95\% confidence interval (CI)
and a 5\% margin of error (MoE)
to estimate a representative sample of feedback statements.
This gives a minimum sample size of 383 statements for GH and KG.
To ensure comprehensive coverage,
we sample from all strata
and ensure that at least one statement is sampled from each stratum.
The sampling is done separately for GitHub and Kaggle.
In total, we obtain 383 statements from GitHub and 433 statements from Kaggle.

\subsection{Data Analysis}

We apply grounded theory and analyze all feedback statements manually.
A code book is developed using a seed of 50 statements,
targetted to the top 5 clusters.
Two members of the research team analyze each statement independently,
to answer ``What the developer is trying to learn or confirm?'' with the statement,
and ``Where in the development lifecycle?'' is the statement defined.
In addition to the statement,
the entire code cell of the statement,
any related code cells where variables or functions are defined,
and the markdown cells containing documentation is analyzed to bring in additional context.

The authors then meet to compare the codes and discuss on disagreements until consensus
is reached on a definition that both can apply consistenly. After two additional iterations,
no new codes are added to the codebook.

The stability of the codebook is verified with a fresh set of 50 statements.
Cohen's kappa score of 0.8 is observed,
which indicates high agreement.
Following this one researcher analyzed and coded all remaining statements.

\subsection{Resampling}

While conducting the manual analysis,
we oberve 91 feedback statements that are not within the scope of this study,
and are excluded from the study.
For instance,
we observe exploratory statements
meant to quickly explore an API
or how an imported function works (EC:EXP),
statements that perform scientific simulations,
but do not train any ML model (EC:NOML),
misuse notebook cells
by writing documentation using block quotes in a code cell
or commented out code with stale output from a prior execution (EC:ERROR),
statements with no functional intent relevant to this study such as \lstinline{plt.ylabel('Win %')} (EC:NOINT),
statements that are not authored in English (EC:ENG),
and finally assertions written in academic notebooks for automatic grading (EC:HW).

To ensure a statistically significant sample size,
we resample replacements from the same clusters where the exclusions occured.
Replacements are verified and if necessary replaced once again.
We perform resampling iteratively until the target sample size (383 for both GH and KG) is reached.

This section presents our findings for the four research questions.
Section~\ref{sec:rq1} establishes what feedback statements practitions author
and how their prevalence varies between GitHub and Kaggle.
In Section~\ref{sec:rq2}, we present the taxonomy of feedback statements
derived from open coding,
and describe the subcodes within each top-level intent.
Section~\ref{sec:rq3}, presents a cross-cutting analysis with the ML pipeline stage
to characterize where each kind of feedback appears
in the development workflow.
Finally, in Section~\ref{sec:rq4}
we relate the feedback taxonomy to the crash taxonomy from Wang et al.\
to surface the duality between defensive practice and observed failure.
A summary of the feedback taxonomy is presented in Table~\ref{tab:taxonomy}.

\paragraph*{Statistical framework}
We test whether two categorical variables are associated,
with the null hypothesis
that the variables are independent.
We adopt a 95\% confidence level
and treat any p-value below 0.05
as a statistically significant association.
We select the test
based on the expected cell frequencies of the contingency table.
% TODO: explicitly state Cochran's rule
When all expected frequencies are at least five
and Cochran's rule holds,
we apply Pearson's chi-squared ($\chi^2$) test.
Otherwise we apply Fisher's exact test
with Monte Carlo simulation ($n=9999$ resamples).
% TODO: describe how to interpret this value
We also report the bias-corrected Cram\'er's $V$ as the effect size for all tests.
Before each test we verify that the validity conditions hold
and we report the outcome by analyzing the residuals that drive the association.
When the marginal distribution of a variable is severely imbalanced,
no test has adequate power and we report descriptive statistics only.

\subsection{RQ1: Prevalence and Variation of Feedback Mechanisms}
\label{sec:rq1}

\highlight{\textbf{Finding 1.} Feedback in ML notebooks is overwhelmingly exploratory.
EXP statements account for 85\% of the corpus.
Formal assertions are rare on Kaggle and common on GitHub.
This platform asymmetry is large and highly significant.}

Our corpus consists of 815 feedback statements
collected from GitHub ($n=382$, 46\%) and Kaggle ($n=433$, 54\%).
The statements split into two top-level intent codes.
EXP accounts for 692 statements (85\%) and VAL accounts for 123 statements (15\%).
The split is highly skewed toward exploratory feedback on both platforms.

\paragraph*{Intent varies sharply by platform.}

We apply $\chi^2$ to test whether intent is associated with notebook source.
The test reveals a significant and large association ($\chi^2(1) = 144.72$, $p < 0.001$, Cram\'er's $V = 0.421$).
Figure~\ref{fig:distribution-intent} shows the distribution.
GH notebooks contain proportionally more formal assertions (31.2\%).
KG notebooks are overwhelmingly exploratory and contain only four assertions (0.9\%).

The asymmetry is consistent with the contrasting purposes of the two platforms.
GitHub hosts software engineering projects
where assertions are a natural practice.
Kaggle hosts data exploration and competition submission workflows
where programmatic validation is uncommon.
Section~\ref{sec:rq3} explores this asymmetry further,
where we show that it interacts with pipeline stage as well.

\paragraph*{Within-intent subcode distributions.}

Each statement is further coded
along ten subcodes that are disjoint across intent.
For EXP statements ($n=692$),
we apply $\chi^2$ to test
whether the subcode is associated with source.
The test reveals a statistically significant but weak association
($\chi^2(3) = 10.77$, $p=0.013$, Cram\'er's $V=0.125$).
For VAL statements ($n=123$),
no statistical test for independence applies reliably,
because the marginal imbalance leaves little power to detect any association.
Figure~\ref{fig:distribution-type} therefore presents
the subcode distribution descriptively for both intents.

Within EXP, four subcodes account for the entire distribution.
INSPECT (44\%) is the most frequent,
followed by COMPUTE (27\%), STRUCT (18\%), and STATS (11\%).
The rank order is consistent with a notebook-as-scratchpad model of feedback.
Practitioners most often display values
to confirm what an expression evaluates to (INSPECT),
compute derived quantities they read off the cell output (COMPUTE),
probe the structure of an object (STRUCT),
and least often request a statistical summary (STATS).
The same rank order holds on both platforms,
with proportions within five percentage points of each other.

Within VAL, the distribution is more skewed.
SHAPE (38\%) and EQ (29\%)
together cover two-thirds of all VAL statements,
followed by BOOL (17\%).
Practitioners predominantly assert dimensional and equality invariants.
Type checks, existence checks, and approximate equality are present
but appear in fewer than one in ten validation statements.
The within-VAL distribution is concentrated almost entirely on GitHub.
Of the 123 VAL statements, 119 occur in GitHub notebooks.
The four Kaggle VAL statements consist of three BOOL and one EQ.
Four of the six VAL subcodes (SHAPE, APPROX, EXIST, TYPE)
do not appear in any Kaggle notebook in the sample.
Kaggle contributors effectively do not use structural validation at all.

\subsection{RQ2: Intent Taxonomy of Feedback Statements}
\label{sec:rq2}

\highlight{\textbf{Finding 2.} The intent taxonomy resolves into ten subcodes
across the EXP and VAL families.
Within VAL, practitioners predominantly check shape, equality, and boolean invariants.
Within EXP, practitioners predominantly inspect, compute, and probe structure.
The subcodes capture distinct concerns
that motivate different tooling responses.}

Open coding of the 815 feedback statements
produces ten subcodes
distributed across the two top-level intents
(see Table~\ref{tab:taxonomy}).
We describe each subcode below
and ground the description in the use cases that emerge from the data.
We organize the description by intent and by subcode frequency.

\paragraph*{Dimensionality check (VAL-SHAPE)}

Shape validation is the most prevalent form of explicit validation in the corpus.
Practitioners use these statements to verify
that arrays and tensors carry the dimensions that downstream operations require.
A shape mismatch in a PyTorch or NumPy pipeline
either raises a cryptic runtime error far from its origin,
or silently produces wrong results through unintended broadcasting.

Shape validation concentrates at three points in the pipeline.
At the model construction stage,
practitioners verify that each layer of a neural network produces the expected output shape
before the next layer is wired in.
Listing~\ref{lst:61} checks the shape of the fifth feature map from a VGG16 encoder
immediately before it is connected to a segmentation decoder.
A similar pattern appears across GAN encoders,
transformer attention blocks,
and GRU sequence models.

\begin{lstlisting}[caption={}, label={lst:61}]
assert output['x5'].size() == torch.Size([BATCH_SIZE, 512, 7, 7])
\end{lstlisting}

At the evaluation stage,
shape validation guards against misaligned prediction and label arrays
before metrics such as RMSE or F1 are computed.
A length mismatch from a dropped remainder batch or an off-by-one error
either crashes the metric computation
or produces a numerically plausible but incorrect score.
A third theme is data integrity after joins, splits, and transformations.
Practitioners assert that a train-test split accounts for all rows,
that a join did not silently add or drop records,
and that a preprocessing step such as \lstinline{QuantileTransformer} preserves the sample count.
These statements reflect an awareness
that pandas operations can silently alter the size of a dataframe
in ways that are not immediately visible in a notebook output.

\paragraph*{Value check (VAL-EQ)}

Practitioners check that a specific value, configuration parameter, or computed result
matches a known expected value.
The expected value is statically known at the time of writing the assertion
and is derived from domain knowledge, architectural constraints, or a prior specification.
Three themes emerge.

The first is the enforcement of architectural constraints
that have no runtime enforcement in the framework.
Sinusoidal positional embeddings require an even embedding dimension
because the vector is split between sine and cosine components (Listing~\ref{lst:4}).
Wide ResNet filter counts must remain even at each resolution level,
and certain ResNet depth configurations must satisfy a modular arithmetic relationship
to accommodate the bottleneck block structure.
Without the assertion,
a violated constraint produces a shape error deep inside the forward pass
whose connection to the misconfigured hyperparameter is not obvious.

\begin{lstlisting}[caption={}, label={lst:4}]
assert dim % 2 == 0
\end{lstlisting}

The second theme is schema validation on loaded data.
Practitioners check that a CSV file has the expected number of classes,
that a required column appears in the correct position,
that an index contains no duplicates,
and that a normalization step produced a zero-sum result.
These checks reflect the reality that data loading involves external files
whose schema cannot be verified statically.

The third theme is the detection of training-serving skew.
Listing~\ref{lst:118} verifies
that the offline preprocessing pipeline and the production preprocessing pipeline
produce identical training data after feature selection.
A divergence between the two would cause the deployed model
to operate on a different feature distribution than the one it was trained on,
which would result in a silent failure with no crash signal.

\begin{lstlisting}[caption={}, label={lst:118}]
assert_frame_equal(X_train_prepro_ks.to_pandas(), X_train_prod_prepro_ks.to_pandas())
\end{lstlisting}

\paragraph*{Boolean invariant (VAL-BOOL)}

Boolean validation captures a wide range of preconditions
that do not reduce to a shape check or an exact equality.
These statements guard pipeline steps against inputs
that are structurally valid but semantically incorrect.

The most common theme is data quality enforcement
before a downstream step that cannot recover from a violated precondition.
Practitioners assert the absence of missing values before feature selection,
the absence of class imbalance before oversampling logic executes,
and the satisfaction of domain constraints before domain-specific computations begin.
Listing~\ref{lst:54} appears in a cardiovascular disease mortality pipeline
to prevent missing values from silently corrupting feature selection
or crashing estimators that assume complete data.

\begin{lstlisting}[caption={}, label={lst:54}]
assert not predictors.isnull().values.any()
\end{lstlisting}

A second theme is the enforcement of geometric and physical invariants
derived from domain knowledge.
An object detection notebook asserts that both coordinates of a bounding box
satisfy the correct ordering before computing intersection over union.
A physics domain notebook asserts that model predictions fall within the interval \([0, 1]\)
as required by the problem formulation.
A word2vec implementation asserts that the number of context words sampled
does not exceed the size of the context window.
A third theme is transfer learning setup validation.
Practitioners assert that all gradient tensors in a frozen backbone are \lstinline{None},
since a backbone that is not fully frozen continues to update during fine-tuning
and silently degrades transfer learning performance without any error.

\paragraph*{Approximate equality (VAL-APPROX)}

Approximate validation appears where exact equality is either mathematically inappropriate
or unachievably strict due to floating-point arithmetic.
Two use cases emerge.
The first is unit testing of numerical implementations.
Practitioners validate custom loss functions, data transformation utilities,
and metric implementations
by comparing their output to a known reference value within a specified tolerance.
Listing~\ref{lst:374} validates
that a custom NDCG implementation returns 1.0 when predicted and true rankings are identical.
The second is reproducibility verification.
Practitioners who reproduce results from a published paper
use approximate assertions to confirm that their experimental setup matches the reported values
within a tolerance that accommodates minor numerical differences
across platforms and library versions
while still catching a substantively incorrect result.

\begin{lstlisting}[caption={}, label={lst:374}]
assert np.isclose(ndcg([3, 1, 2], [3, 1, 2], 3), 1)
\end{lstlisting}

\paragraph*{Existence check (VAL-EXIST)}

Existence validation checks for membership in or presence within a dynamic runtime structure.
The first theme is data leakage prevention.
Listing~\ref{lst:84} verifies that training and validation sets are disjoint
before model fitting begins.
Data leakage from validation into training inflates evaluation metrics
without any runtime error,
and the only way to detect it programmatically is to check for overlap explicitly.

\begin{lstlisting}[caption={}, label={lst:84}]
assert set(train_dataset.filenames).isdisjoint(set(valid_dataset.filenames))
\end{lstlisting}

The second theme is precondition checking on external resources.
Practitioners assert that a required file path exists before attempting to load it,
and that a hyperparameter name is present in the parameter space of an estimator
before launching an expensive grid search.
In both cases the failure mode without the assertion
is either a cryptic I/O error
or a silent search over the wrong parameter space.

\paragraph*{Type check (VAL-TYPE)}

Practitioners use these statements to enforce
that objects carry the correct Python type or NumPy dtype
before passing them to operations that would fail silently or with a misleading error
if given the wrong type.
A common pattern asserts that model parameters are NumPy arrays rather than Python lists,
since many NumPy operations accept both but produce different results.
The low frequency of VAL-TYPE likely reflects the fact
that Python's dynamic typing and framework-level type coercion
often suppress type errors silently.
When a wrong type does not immediately crash the computation,
practitioners have no obvious signal that a check is needed.

\paragraph*{Object inspection (EXP-INSPECT)}

Inspection statements are the most frequent form of exploratory feedback in the corpus.
A practitioner places a bare variable name, a plot call,
or a lightweight accessor expression as the last statement in a cell
to surface the current state of a data structure without transforming it.
The output is consumed visually and immediately.

Inspection is used throughout the pipeline
but concentrates at the data loading and preparation stages,
where practitioners establish a running mental model of the dataset
before any transformation is applied.
A bare \lstinline{.head()} call after loading a CSV (Listing~\ref{lst:209})
confirms that column names parsed correctly,
that data types look plausible,
and that no obvious loading artefact is present.
The same lightweight pattern monitors the effect of individual cleaning operations,
such as inspecting a column after imputing missing values to confirm that the fill worked.

\begin{lstlisting}[caption={}, label={lst:209}]
df_subjects.head()
\end{lstlisting}

At the evaluation stage,
inspection shifts from data structure verification to model output monitoring.
Practitioners surface intermediate predictions, confusion matrices, learned coefficients,
and cross-validation results as bare last-cell expressions.
During training, a recurring pattern is the periodic visual inspection of generated outputs
or loss curves.
One notebook displays a GAN-generated image at regular intervals
to assess whether the generator is producing realistic samples.
Another plots the training log-likelihood over epochs to monitor convergence.
These statements serve as progress signals
that are cheaper and more immediate than a formal metric computation.

A theme distinctive to GH notebooks is the use of inspection to debug internal model state.
Practitioners surface the names of specific layers in a neural network,
the values of learned weight slices,
or the output of a forward hook to understand what the model is doing internally.
Inspection of this kind is not possible with a high-level metric,
as it requires looking directly at model internals.

\paragraph*{Exploratory computation (EXP-COMPUTE)}

Compute statements execute a transformation, aggregation, or model operation
and return a result that the practitioner examines.
The distinction from inspection is that the result is not already present in a variable
but is produced on the fly by the statement itself.

The dominant use of EXP-COMPUTE in both GH and KG notebooks
is metric computation at the evaluation stage.
Practitioners compute accuracy, RMSE, F1, NDCG, AUC, and log-loss
as last-cell expressions
to obtain an immediate reading of model performance (Listing~\ref{lst:339}).
These calls are repeated across multiple experimental runs
to drive an iterative model and hyperparameter selection process.

\begin{lstlisting}[caption={}, label={lst:339}]
accuracy_score(test_y, rf_pred_y)
\end{lstlisting}

A second theme is informal postcondition checking on preprocessing steps.
After applying a standard scaler,
practitioners compute the mean and standard deviation of the transformed features
to verify that both are near their expected values.
After a \lstinline{dropna} operation,
practitioners compute a null count to confirm that missing values were removed.
These checks express an expectation about the output of the preceding operation
but rely on visual judgment rather than a programmatic threshold.

A third theme, distinctive to GH notebooks,
is the use of \lstinline{model.fit()} calls as pipeline integrity checks.
Practitioners run a short training call with a fraction of the data
to confirm that the pipeline runs without error
and that the loss decreases
before committing to a full training run.

In KG notebooks, EXP-COMPUTE is also used extensively for exploratory feature analysis.
Practitioners compute grouped aggregations and correlation matrices
to understand whether a candidate feature has a meaningful relationship with the target
before deciding whether to include it.
A grouped statistic such as a survival rate by passenger class and sex
motivates a subsequent feature engineering step in the next cell.
This compute-then-act pattern is a core workflow unit in competition notebooks.

\begin{lstlisting}[caption={}, label={lst:480}]
test_raw.shape[0] == test.shape[0]
\end{lstlisting}

Listing~\ref{lst:480} appears in a KG notebook as a last-cell boolean expression
that verifies no rows were lost during a transformation.
The expression is not wrapped in an \lstinline{assert},
so a False result is visible in the cell output but does not halt execution.
The intent is identical to a shape assertion,
which we return to in Section~\ref{sec:implicit-validation}.

\paragraph*{Structural probe (EXP-STRUCT)}

Structural inspection statements query the metadata of a data structure rather than its values.
They surface shape, column names, dtypes, unique values, cardinality, or value counts.
EXP-STRUCT concentrates overwhelmingly at the preparation stage (78 of 126 statements),
where practitioners use these lightweight queries
to understand and verify the current state of the data
before applying the next transformation.

A pervasive theme is the use of structural inspection
as a lightweight checkpoint after each cleaning or encoding step.
After dropping columns, a practitioner inspects \lstinline{.columns} to confirm the drop worked.
After one-hot encoding, a practitioner checks \lstinline{.shape}
to confirm that the column count increased as expected.
After undersampling the majority class,
a practitioner inspects the shape of the resulting arrays
to confirm that the class ratio changed (Listing~\ref{lst:593}).
These checkpoints serve the same purpose as postcondition assertions
but rely on visual confirmation rather than programmatic enforcement.

\begin{lstlisting}[caption={}, label={lst:593}]
(X_train_under.shape, y_train_under.shape, X_test_under.shape)
\end{lstlisting}

A second theme is the use of \lstinline{.unique()} and \lstinline{.value_counts()}
to audit the actual contents of a categorical column before deciding on an encoding strategy.
Practitioners check whether unexpected values or label inconsistencies are present
before applying \lstinline{LabelEncoder} or \lstinline{OneHotEncoder}.
Without this inspection,
the encoder would treat surface variants of the same category as distinct
and produce a semantically incorrect feature.

A third theme, more common in GH notebooks,
is the use of structural queries to set downstream model parameters dynamically.
Practitioners inspect \lstinline{.vector_size} of a word embedding
to determine the input layer dimension of a neural network,
or check \lstinline{X_train.shape[2]}
to set the \lstinline{input_shape} parameter of a 1D CNN.
These statements serve as interface checks
between the data preparation and model construction stages.

\paragraph*{Statistical summary (EXP-STATS)}

Statistical summary statements produce distributional views of a feature or target variable.
They include calls to \lstinline{.describe()}, \lstinline{.hist()}, \lstinline{.value_counts()},
and distribution or box plots via seaborn and matplotlib.
EXP-STATS is proportionally more prevalent in KG notebooks (13.5\% of KG EXP)
than in GH notebooks (7.6\% of GH EXP),
which reflects Kaggle's culture of systematic exploratory data analysis before modelling.

The dominant theme is pre-modelling data auditing.
Practitioners use distribution plots and summary statistics
to detect outliers, assess class imbalance,
and check whether a feature's distribution motivates a transformation.
A seaborn \lstinline{distplot} of insurance charges (Listing~\ref{lst:634})
reveals a right-skewed distribution
that motivates a log transform before fitting a linear regression model
whose residuals are assumed to be normally distributed.
A box plot of engine power values in a used car dataset
drives a decision to remove extreme outliers.
A histogram of model prediction confidence scores
reveals whether the model is well-calibrated.
In each case the statistical summary is not an end in itself
but a decision point that gates the next preprocessing or modelling step.

\begin{lstlisting}[caption={}, label={lst:634}]
sns.distplot(data.charges)
\end{lstlisting}

A second theme is post-processing verification.
After scaling features with \lstinline{StandardScaler},
practitioners call \lstinline{.describe()}
to confirm that the mean is near zero and the standard deviation is near one.
After imputing missing values,
a practitioner plots the column's distribution
to confirm that the imputed values do not distort the shape
in a physiologically implausible direction.
These verification uses of EXP-STATS
occupy the same functional niche as EXP-STRUCT checkpoints,
but do so through distributional inspection rather than structural metadata.

A third theme, present primarily in GH notebooks,
is the use of statistical summaries to diagnose model behaviour.
A histogram of a model's confidence scores across a held-out test set
reveals whether predictions cluster near 0.5 or toward 0 and 1.
A KDE plot of SARIMA residuals checks whether the model's errors
approximate a normal distribution,
which is a formal assumption of the ARIMA model class.
These uses treat the statistical summary as a model diagnostic tool
rather than a data auditing tool.

\subsection{RQ3: Pipeline Stage Distribution of Feedback Statements}
\label{sec:rq3}

\highlight{\textbf{Finding 3.} Feedback concentrates in different pipeline stages on each platform.
Kaggle notebooks concentrate on data preparation.
GitHub notebooks distribute feedback across data ingestion, training, and evaluation.
Within GitHub, intent is associated with stage,
but the choice of subcode within each intent is largely stage-agnostic.}

This section examines where in the ML pipeline
practitioners author feedback statements,
and how the choice of intent and subcode varies by stage.
We organize the analysis as a sequence of three cross-tabulations
of increasing granularity.
First, we examine stage by source
to characterize platform-level differences in where feedback appears.
Next, we examine intent by stage,
stratified by source to control for the platform-level confound established in RQ1.
Finally, we examine subcode by stage on the GitHub subset only,
because Kaggle's heavy concentration in data preparation
and its small VAL count
preclude meaningful comparisons.

\paragraph*{Stage distribution and platform asymmetry.}

PREP (40\%) is the single largest stage,
followed by EVAL (21\%), DATA (12\%), FEAT (12\%), TRAIN (9\%) and MODEL (6\%).
The corpus contains 2 statements (0.25\%) where the stage was unclear (UNC).
Notebook feedback is concentrated upstream of model training, not downstream.

We apply the $\chi^2$ test
to examine whether ML pipeline stage is independent of notebook source.
The UNC stage ($N=2$) is excluded which leaves $N=813$ statements across six stages.

The test indicates a highly significant and large association
($\chi^2(5) = 158.04$, $p<0.001$, Cram\'er's $V=0.441$).
Figure~\ref{fig:distribution-stage} shows the distribution.
Kaggle notebooks concentrate predominantly on the PREP stage (58\%),
while GitHub notebooks distribute more evenly
across DATA (19\%), EVAL (25\%) and TRAIN (13\%).
The MODEL stage is almost entirely absent on Kaggle (0.5\% vs. 11\% in GitHub).

The dominance of the PREP stage and absence of the MODEL stage in Kaggle notebooks,
reflects the competition format.
Competition notebooks concentrate on data preparation pipelines
and typically consume pre-configured model APIs
rather than defining model architectures.

\paragraph*{Intent varies with stage on GitHub.}

We examine if the intent of statements is associated with the stage in which they occur.
We stratify the analysis by source,
since source is associated with both intent and stage
and thus acts as a confounding variable.
For Kaggle statements ($n=433$),
we apply Fisher's exact test
which reveals a statistically significant association
($p=0.009$, Cram\'er's $V=0.364$).
The test is valid under these conditions
but rests on only 4 VAL statements,
which makes the effect size estimate unstable at such low counts.
We therefore restrict further analysis to GitHub statements ($n=380$).

We apply $\chi^2$ to examine whether intent is associated with pipeline stage on GitHub.
The test reveals a statistically significant association
($\chi^2(5)=20.34$, $p=0.001$, Cram\'er's $V=0.231$).
Figure~\ref{fig:distribution-stage-intent} shows that the VAL share varies by stage.
MODEL is the most validation-heavy stage (60\%),
followed by TRAIN (34\%) and FEAT (29\%).
DATA (76\% EXP) and EVAL (77\% EXP) are more exploratory.
Practitioners validate when the code defines reusable structure,
such as model definitions and training loops,
and inspect when the code is exploratory and data-facing,
such as preprocessing and evaluation.

\paragraph*{Within each intent, subcode is largely stage-agnostic.}

\begin{figure}
  \begin{center}
    \includegraphics[width=\textwidth]{distribution-stage-type.pdf}
  \end{center}
  \caption{Subcode distribution by stage on GitHub statements.
  Within EXP, COMPUTE peaks in TRAIN and EVAL, STATS in PREP and DATA,
  AND INSPECT dominates every stage.
  Within VAL, SHAPE is the modal subcode in four of six stages,
  and EQ appears at every stage.}
  \label{fig:distribution-stage-type}
\end{figure}

We restrict the subcode-stage analysis to GitHub for the same reason as above.
Kaggle contains too few VAL statements to support inference,
and Kaggle's heavy concentration in PREP obscures within-stage subcode patterns.
We report the GitHub distribution descriptively
since the substantive question is which feedback subcodes appear at which stages,
not whether the two variables are statistically dependent.

Within the GitHub EXP statements ($n=263$),
INSPECT is the dominant exploratory subcode at every stage
and accounts for 36\% to 57\% of EXP within each stage.
Three patterns stand out for the remaining subcodes.
COMPUTE peaks in TRAIN (52\%) and EVAL (47\%),
where practitioners compute losses, predictions, and metrics inline.
STATS appears almost exclusively in PREP (19\%) and DATA (9\%)
and is absent from FEAT and TRAIN.
This matches the role of summary statistics in early data understanding.
STRUCT is small but spread across DATA (20\%), FEAT (23\%), MODEL (25\%), and PREP (17\%),
reflecting structural probing of inputs, engineered features, and model components.

Within the GitHub VAL statements ($n=117$), the picture differs.
The choice of validation subcode is largely stage-agnostic.
SHAPE is the modal validation subcode in four of the six stages
(DATA, EVAL, MODEL, PREP)
and accounts for 39\% to 55\% of VAL within those stages.
EQ is modal in FEAT (50\%) and TRAIN (29\%)
and appears at every stage.
BOOL appears at every stage in small numbers (4\% to 26\%).
The rarer subcodes (APPROX, EXIST, TYPE) appear too few times
to support stage-level claims,
with the single exception of EXIST in TRAIN
where four of the six EXIST statements occur.

The above patterns together show that
whether a statement is validation or exploration depends on stage,
but within each intent the choice of subcode does not.
Practitioners pick exploratory subcodes
according to what each stage demands of them
and pick validation subcodes
according to what each statement asserts about,
not according to where in the pipeline the assertion appears.

RQ1 and RQ2 characterize what feedback statements practitioners author,
and RQ3 shows where they author them.
RQ4 situates those findings against the failure modes
identified by analyzing crashes in Jupyter notebooks.
Wang et al.~\cite{wang2025why} examined ML Jupyter notebooks
collected from GitHub and Kaggle,
and cataloged \emph{crashes}
which halt execution and surface an exception in the cell output.
While the Wang taxonomy identifies the failure mode,
our taxonomy provides a complementary view,
by identifying validation statements
designed to pre-emptively catch such errors.

To make this duality concrete,
we map each VAL subcode to the crash types it pre-empts.
Next, we compare the stage distribution of feedback statements
to the stage distribution of crashes.
Additionally, we identify defensive practices in our corpus
that crash analysis cannot observe
because the practices exist precisely to prevent silent failures.

\paragraph*{Mapping VAL subcodes to crash types.}

We construct the mapping in Table~\ref{tab:wang-mapping}
by checking, for each VAL subcode,
whether an assertion either prevents the crash from occurring
or surfaces an upstream defect
that would otherwise produce that crash type at a later stage.

VAL-SHAPE maps to three of the six ML-specific crash types Wang et al.\ identify.
Tensor Shape Mismatch (5.5\%), Unsupported Broadcast (1.6\%), and Feature Name Mismatch (1.2\%)
together account for 8.3\% of crashes in their sample.
Our results are consistent with this prevalence.
VAL-SHAPE is the most prominent validation subcode in our corpus
and accounts for 38.84\% of VAL statements.

VAL-EQ and VAL-BOOL pre-empt crashes that arise
when downstream code receives semantically incorrect inputs.
Wang et al.\ report that API misuse alone accounts for
66\% of Invalid Argument crashes and 31\% of Data Value Violation crashes.
Architectural constraint assertions (Listing~\ref{lst:4})
pre-empt Model Initialization Error,
and data quality checks (Listing~\ref{lst:54})
pre-empt Data Value Violation crashes.
The latter directly mirrors the data confusion root cause identified by Wang et al.

Wang et al.\ attribute Variable Not Found crashes
primarily to notebook-specific issues such as out-of-order execution,
which a code-level assertion cannot prevent.
However, a subset of these crashes
arise from references to objects
that should have been constructed earlier in the same cell
or earlier in a logically ordered sequence of cells.
VAL-EXIST statements that check for the presence of a key, attribute, or filename
before the dependent operation runs
catch this subset of Variable Not Found crashes
and surface the missing dependency as an explicit assertion failure
rather than a downstream cryptic error.

VAL-TYPE pre-empts the API-misuse subset of type-related crashes.
Wang et al.\ report that 42\% of Type Error crashes
are caused by API misuse,
and that 44\% of Attribute Error crashes share the same root cause.
VAL-TYPE statements (e.g., \lstinline{assert type(embedding_layer) == Embedding})
enforce the type contract that the API expects
and convert a downstream cryptic error into an explicit assertion failure.

\paragraph*{Crashes and validations cover different parts of the failure space.}

Wang et al.\ report
Out of Memory, Module Not Found, IO Error, Environment Error, and Request Error crash types
that together account for 28\% of crashes in the Wang sample.
These crash types however are not captured by our taxonomy of validation statements
because these crashes are not preventable through code-level assertions.

In contrast,
our taxonomy captures defensive practices
that are invisible to a crash-based study.
For instance,
VAL-APPROX has no direct counterpart in the Wang taxonomy.
Approximate equality assertions verify reproducibility
and catch large drifts in the results
across multiple pipeline runs
and different data versions.
For instance,
Listing~\ref{lst:374} pre-emptively stops the execution
when the NDCG diverges from the expected value.
Without the assertion,
the pipeline would execute and produce an incorrect score,
without raising an exception in the cell output.
Similarly, the training-serving skew detection theme within VAL-EQ (Listing~\ref{lst:118})
catches a divergence that produces no crash signal at all.
The data leakage prevention theme within VAL-EXIST (Listing~\ref{lst:84})
catches a defect that inflates evaluation metrics
without producing any error output.
A study that examines only crashes,
therefore cannot observe these practices
because the practices exist precisely to prevent silent failures
that never reach the cell output as an exception.
The asymmetry suggests that the failure space of ML notebooks
is wider than what crash analysis alone can capture.

\paragraph*{Convergence on data preparation.}

Despite the inverted lenses,
the two studies converge on data preparation
as the empirical centre of gravity
in ML notebook development.
Wang et al.\ measure where execution failes,
and report that 33.0\% of crashes occur during data preparation.
We measure where practitioners author feedback statements,
and independently report
that 40\% of feedback statements occur during the same stage.
The convergence of these two independent studies,
suggests that data preparation stage
demands the most attention and effort from pracitioners to handle crashes,
and consequently the largest share of tooling investment.

The platform asymmetry from RQ1 and RQ3 is also consistent across both studies.
Wang et al.\ report that GitHub notebooks contain four times more crashes than Kaggle (10.67\% vs.\ 2.61\%),
and that the crash distribution across pipeline stages
diverges between the two platforms in the same direction we observe for feedback.

\section{Discussion}

\subsection{Practical Implications}
\label{sec:practical-implications}

\subsubsection{Notebooks as scratchpads, not test harnesses}
\label{sec:scratchpad}

Our results show that 85\% of feedback statements in ML notebooks are exploratory,
which indicates that practitioners read values off the cell output
rather than enforce them programmatically.
This pattern holds on both platforms
but is most extreme on Kaggle,
where formal assertions account for fewer than one in a hundred feedback statements.
The dominant feedback mode in ML notebook development
is therefore visual confirmation, not assertion-based validation.

Breck et al.~\cite{breck2017ml} argue
that training data deserves the same testing discipline as production code,
and propose a rubric for assessing the readiness of an ML pipeline.
The same authors later introduce \emph{TensorFlow Data Validation}~\cite{breck2019data},
a tool that validates each batch of data against a declared schema.
Data validation libraries such as TFDV and Great Expectations
require practitioners to author the schema as a separate artefact.
Our results indicate that
practitioners already author the same checks in their notebooks,
but in an exploratory rather than declarative form.

We see two concrete directions for tool developers
that amplify what practitioners already do.
First,
EXP-STRUCT statements
such as \lstinline{df.shape} and \lstinline{X_train_under.shape}
serve as informal post-condition checkpoints after data transformations.
A static analyser can detect these statements,
record the values they produce on a successful run,
and propose a corresponding shape assertion.
or a shape constraint in a data schema.
The taxonomy proposed in Section~\ref{sec:rq3}
can be used to identify candidate assertions
that were authored by practitioners as structural probes.

Second,
EXP-STATS calls to \lstinline{.describe()}
that follow a scaling or imputation step
encode an expected post-condition (e.g., mean near zero, no missing values).
A tool can extract these expectations from the visible output
and emit an explicit precondition for the next cell,
or contribute the same expectation
to a Great Expectations suite attached to the notebook.

\highlight{\textbf{Takeaway 1.}
ML notebook feedback is predominantly visual, not programmatic.
Tooling that treats exploratory statements as first-class feedback signals
covers a much larger surface area
than assertion-based tooling alone.}

\subsubsection{Implicit validation through last-cell expressions}
\label{sec:implicit-validation}

Within the EXP-COMPUTE category,
practitioners author boolean expressions in the last cell of a code block
that are structurally exploratory
but functionally equivalent to assertions.
For instance,
Listing~\ref{lst:480} verifies that no rows were lost during a transformation.
Similarly,
the statement \lstinline{pd.Series(options_short_list).isin(dummies.columns).all()}
verifies that all expected columns are present after one-hot encoding.
Since neither expression is wrapped in an \lstinline{assert},
a False result is visible in the cell output
but does not halt execution.

The observed pattern
bridges the boundary between exploratory and formal validation statements,
and reveals a class of latent assertions
already present in notebooks.
Each last-cell boolean expression that evaluates to True on a successful run
is a candidate that can be promoted to a formal assertion
without any change in practitioner intent.
This can be done automatically by a static analyzer
that identifies last-cell expressions
whose root AST node is a comparison or boolean operation.
The tool can record the expression's value across executions of the notebook,
and surface candidates whose value is consistently True.

Since these statements produces visible output but does not halt execution,
downstream cells in the notebook
continue to run on data
that violates an assumption
the practitioner authored and enforced manually.
The decision to proceed despite the violation
is therefore implicit and undocumented,
and accumulates as technical debt
as data, personnel, and execution environments change
over the lifetime of the notebook~\cite{sculley2015hidden,sambasivan2021everyone,pimentel2019large-scale}.
Promoting the same expression to an \lstinline{assert}
makes the assumption explicit
and turns an undocumented decision into a checked invariant.

The promotion also addresses a reproducibility threat
that is specific to the notebook format.
Notebook cells can execute in any order,
and an out-of-order execution
can leave the kernel in a state
that does not reflect the source code~\cite{wang2020assessing,pimentel2019large-scale}.
A last-cell boolean expression
evaluates against the kernel state at the moment of execution
but does not offer protection against a later out-of-order run.
An \lstinline{assert} authored at the same point
fires every time the cell is re-executed,
regardless of the order in which surrounding cells were run.

\highlight{\textbf{Takeaway 2.}
Practitioners author validation logic in the form of last-cell boolean expressions
that are structurally exploratory but functionally equivalent to assertions.
Static analysers can detect these expressions
and propose their promotion to formal assertions
without changing practitioner intent.}

\subsubsection{Platform asymmetry}
\label{sec:platform-asymmetry}

The two platforms in our study host two qualitatively different modes of ML work.
Kaggle notebooks concentrate 58\% of feedback in the data preparation stage
and contain almost no model-construction code.
GitHub notebooks distribute feedback across data ingestion (18.7\%),
evaluation (24.7\%), training (13.2\%), and model construction (11\%).
The MODEL stage is almost entirely absent on Kaggle (0.5\%).
Validation effort follows the same asymmetry,
as GitHub contains 119 of the 123 VAL statements in our corpus.

Kaggle competition workflows operate primarily on data exploration,
and delegate evaluation of the model to the platform leaderboard.
GitHub on the other hand, hosts a broader range of activities that spans the entire ML development lifecycle.
The Wang et al.\ study reports a similar asymmetry
where GitHub notebooks contain four times more crashes compared to Kaggle (10.67\% vs.\ 2.61\%),
and the crash distributions across pipeline stages
diverge similar to our observations for feedback statements.
Our findings highlight the difference in the kind of work each platform hosts,
and show that the platforms are not interchangeable data sources.

\highlight{\textbf{Takeaway 3.}
GitHub and Kaggle host qualitatively different modes of ML work.
Studies of ML developer practice
that draw on a single source systematically under-sample
the practices concentrated on the other.
Source should be treated as a confounder by default.}

\section{Threats to Validity}
\label{sec:threats}

We discuss threats to the validity of our findings
along three dimensions: external, internal, and construct.

\paragraph*{External Validity}

\textbf{Data selection bias.}
Our corpus consists of publicly shared notebooks from GitHub and Kaggle.
This excludes notebooks that practitioners author privately
and notebooks that are part of proprietary or industrial workflows.
Feedback practices in industrial ML notebooks may differ from public ones,
particularly where production handoffs, code review, or compliance requirements
introduce stricter validation discipline.
Our findings therefore reflect the feedback practices visible in public notebooks
and may not generalise to industrial settings.
A study of industrial ML prototype notebooks would be required
to assess how far our findings extend.

\textbf{Temporal scope.}
The GitHub data is collected through GitHub's code search on a single date in June 2023.
The Kaggle data is drawn from KGTorrent,
% TODO: confirm these dates
which was collected between November--December 2021.
Notebook practices evolve as libraries, tooling, and platform features change.
Patterns observed in our corpus may not reflect the most recent conventions.
We mitigate this threat by collecting notebooks across a broad time window
and by focusing on patterns rather than specific API usage.
Replication on a more recent corpus is the appropriate way to test temporal stability.

\textbf{Library scope.}
We restrict the corpus to notebooks that import at least one of four ML libraries:
Scikit Learn, PyTorch, TensorFlow, or Keras.
This excludes notebooks that use alternative ML libraries such as
JAX, XGBoost, or LightGBM as their primary modelling tool.
Prior work~\cite{psallidas2019data} examined six million GitHub Jupyter notebooks
and confirmed that most ML notebooks use the four libraries we include,
which limits but does not eliminate the threat.
The exclusion of Pandas, NumPy, and SciPy as standalone qualifiers is deliberate
and is necessary to avoid conflating data science notebooks with ML notebooks.

\textbf{Platform asymmetry.}
GitHub and Kaggle host different modes of ML work
and produce qualitatively different feedback patterns.
Findings derived from GitHub may not transfer to Kaggle and vice versa.
We address this directly in the analysis by stratifying every test by source
and reporting platform-level differences.

\paragraph*{Internal Validity}

\textbf{Last-cell expression and output drift.}
Our extraction of last-cell expressions
assumes that the source code in the cell matches the output stored in the notebook.
A practitioner who edits a cell after execution,
without re-running it,
leaves the cell in an inconsistent state.
The expression we extract
can then differ
from the expression that produced the output.
We mitigate this threat
by analysing the static intent of the expression
rather than its runtime value.
Our taxonomy is thus anchored in what the practitioner authored,
not what was last executed.
For instance,
a statement coded as EXP-STRUCT
remains a structural probe
regardless of whether the displayed output matches the current code.
The threat is therefore confined
to cases where the static intent itself is ambiguous,
which the codebook resolution process is designed to handle.

\textbf{Manual coding and inter-rater reliability.}
Our analysis depends on manual coding of feedback statements,
which introduces subjectivity,
and the risk of inconsistent application of codes.
We mitigate this through several measures.
Two researchers independently code an initial seed of 50 statements
and Cohen's kappa is reported as a measure of inter-rater agreement.
The codebook is then validated on a fresh set of 50 statements
and further iterations are performed
until consensus is reached on the definition of each code.
The remaining statements
are coded by a single researcher after stabilization,
which carry a residual risk,
since these statements are not coded by both researchers.

Spot-checks during the analysis surface no systematic drift,
but a complete double-coding of the corpus is the most rigorous remedy
and is left to future replication work.

\textbf{Codebook saturation.}
Saturation is reached
when no new codes are introduced
across two consecutive iterations of the codebook.
We meet this criterion within three iterations.
However,
saturation is a property of the sample,
not the population.
Therefore,
it is possible that rare patterns exist in the broader population
that our sample does not surface.
The proportional stratified sampling design (Section~\ref{sec:sampling})
is intended to mitigate this threat
by ensuring that each high-density semantic cluster contributes at least one statement to the sample.
The residual stratum of merged small clusters bounds the effect to clusters
that individually account for fewer than 1\% of the population.

\textbf{Primary intent rule for compound statements.}
Some assertions chain multiple checks within a single statement
(e.g., \lstinline{assert len(top_negative_words) == 25 and [isinstance(w, str) for w in top_negative_words]}
which combines a length check with an element-wise type check).
Similarly,
some exploratory statements chain a transformation (e.g., \lstinline{.groupby()})
with a structural probe (e.g., \lstinline{.head()}).
For such cases,
we adopt the primary intent rule
and code each compound statement under the dominant check.
This compresses the type distribution toward the dominant types
and may under represent secondary types within compound statements.
The effect is small in practice however,
as compound statements account for fewer than 5\% of the corpus.

\textbf{Stage assignment.}
Pipeline stage is assigned
by tracing the role of the statement
in the surrounding cell and notebook context.
Assignment is unambiguous in most cases
but can be uncertain
when a single cell contains code spanning multiple stages.
We resolve such cases through the codebook rule
that assigns a statement
to the stage of the construct it operates on,
and give priority to the most upstream stage
when the construct spans several.
Two statements (0.25\% of the corpus) are coded as UNC because no rule could be applied.
The small share of UNC statements
indicates that the rule is broadly applicable.

\paragraph*{Construct Validity}

\textbf{Definition of feedback.}
We define feedback
as a combination of assertion statements,
and last-cell expressions that produce visible output.
This excludes feedback delivered through other channels
such as inline print statements that are not in the last position,
logging calls,
and visualisations rendered through magic commands~\footnote{Built-in
magic commands in IPython: https://ipython.readthedocs.io/en/stable/interactive/magics.html}.
Our choice is grounded in the practitioner workflow,
since last-cell expressions are the canonical idiom for inspecting state in Jupyter
and assertions are the canonical idiom for enforcing invariants.
This scope may nevertheless under-represent practitioners
who prefer print-based or log-based feedback.
Our corpus does not exhaust the space of feedback in ML notebooks,
but it captures the two dominant idioms.

\textbf{Cluster structure and sample composition.}
We cluster statements using semantic embeddings from CodeBERT
and use the resulting clusters as strata for proportional sampling.
The clustering serves only as a sampling device to ensure diversity.
Classification is done through manual coding,
that determines the intent of each sampled statement,
independently of the cluster it came from.

\textbf{Mapping to the Wang taxonomy.}
The mapping in Section~\ref{sec:wang-comparison}
is established on functional grounds,
by judging whether an assertion would prevent
or surface a given crash type.
The judgement is informed by the Wang root cause analysis
but is itself a construction of this study.
Different judgements are possible at the boundaries,
particularly for the VAL-EXIST to Variable Not Found mapping
which depends on whether the missing reference is a within-cell defect
or a notebook-semantic defect.

We document the mapping rule explicitly
to allow replication and disagreement.

\section{Related Work}

We position our study against three bodies of prior work.
First, empirical studies of bugs and crashes in ML notebooks,
which characterize the failure modes that we treat as the dual of practitioner-authored validation.
Second, large-scale mining and analysis of computational notebooks,
which establishes the methodological foundation for working with notebook corpora at scale.
Third, validation and testing practice in ML and software engineering more broadly,
which provides the conceptual grounding for our intent dimension.

\subsection{Bugs and Crashes in ML Notebooks}
\label{sec:related-bugs}

Wang et al.~\cite{wang2025why} present the closest prior study to ours.
They catalog 92,542 crashes across 64,031 ML notebooks
collected from GitHub and Kaggle
and classify the crashes based on their
type, root cause, ML pipeline stage, and the ML library responsible.
The two studies differ in what they observe.
Wang et al.\ examine error outputs that surface as exceptions in the cell.
In contrast,
we examine assertions and last-cell expressions
that practitioners write
% TODO: an assertion will raise an AssertError exception
to confirm that no exception will surface.

De Santana et al.~\cite{desantana2024bug} classify general bugs in Jupyter notebooks
into eight categories,
focusing on environment setup, kernel, notebook conversion, and implementation issues.
Their study draws from Stack Overflow posts, GitHub commits, interviews, and surveys
rather than from the cell outputs of executed notebooks.
The category breakdown reflects this difference.
Environment and settings issues dominate their data
because the underlying notebooks are predominantly demonstrations and documentation
authored by ML library maintainers,
where commits update dependencies rather than fix implementation defects.
Our study draws from notebooks authored by ML library users
and observes a different distribution of practice.
Together with Wang et al.,
the two studies establish the failure side of ML notebook development.
Our study addresses the success side.

Jiang et al.~\cite{jiang2025exploring} conduct a contemporary empirical study
of bugs and vulnerabilities across the Jupyter ecosystem.
They analyse 1000 notebooks from GitHub
filtered by repository activity and project type,
and apply clustering followed by stratified manual analysis
in a methodology closely related to ours.
Their focus is on bug categories at the project level
rather than feedback mechanisms at the statement level,
which makes their findings complementary to ours.

A separate line of work targets specific defect classes in notebooks.
Patra and Pradel~\cite{patra2022nalin}
present \textsc{Nalin},
a technique that detects name-value inconsistencies
where a variable's name suggests one type or shape
but its runtime value carries another.
This is one of the few prior efforts that operates on the same kind of artefact
that we study (variables visible in the cell context)
and that targets the same class of latent defect
that VAL-TYPE assertions are designed to catch.
\textsc{Nalin} is a runtime detection technique
whereas our study is an empirical characterisation of practitioner-authored checks.

\subsection{Mining and Analysis of Computational Notebooks}
\label{sec:related-mining}

Large-scale mining of Jupyter notebooks has become an established methodology.
Pimentel et al.~\cite{pimentel2019large-scale} mine 1.4 million notebooks from GitHub
to study quality and reproducibility,
and propose guidelines for improving notebook reproducibility.
Psallidas et al.~\cite{psallidas2019data} analyse six million notebooks,
two million enterprise data science pipelines,
and metadata from twelve major data science libraries
to characterise the evolving landscape of computational notebook usage.
Quaranta et al.~\cite{quaranta2021kgtorrent} construct \textsc{KGTorrent},
a public corpus of approximately 250,000 Python notebooks from Kaggle
together with a relational database of metadata.
We use \textsc{KGTorrent} as our Kaggle data source
and follow the broader filtering pattern established by Pimentel et al.\ for the GitHub corpus.

Two recent contributions extend this foundation.
Ghahfarokhi et al.~\cite{ghahfarokhi2024distilkaggle}
present \textsc{DistilKaggle},
a curated dataset of Kaggle notebooks
with finer-grained metadata than \textsc{KGTorrent}.
Grotov et al.~\cite{grotov2022large-scale} compare
code quality between Jupyter notebooks and Python scripts on GitHub.
Their finding that notebooks systematically deviate from script-style coding conventions
is the broader pattern in which our platform asymmetry result sits.

A complementary methodological contribution comes from Jiang et al.~\cite{jiang2022evolution}
who label notebook cells with their position in the ML development lifecycle
through static data flow analysis.
Their stage taxonomy informs the pipeline stage dimension we use in our coding.
We adopt a similar set of stages
(data ingestion, preprocessing, feature engineering, model construction, training, evaluation)
but apply them at the statement level rather than the cell level
and assign stage through manual coding rather than static analysis.

\subsection{Validation and Testing Practice}
\label{sec:related-testing}

Empirical work on testing and validation practice in software engineering
has examined how developers use assertions and lightweight checks.
Kochhar et al.~\cite{kochhar2017revisiting} replicate and extend a prior study
of assertion usage on 185 Java projects from GitHub,
and report that assertion adoption remains low despite established benefits.
Vidoni~\cite{vidoni2021evaluating} studies unit testing practices in 177 R packages
and identifies systematic gaps in test coverage
together with sources of testing technical debt.
Both studies establish a baseline.
Programmatic validation is under-adopted across language ecosystems.
Our finding that formal assertions account for 15\% of feedback statements
on GitHub and under 1\% on Kaggle
extends this baseline to the ML notebook setting.

In the ML domain specifically,
two prior taxonomies bear directly on our work.
Humbatova et al.~\cite{humbatova2020taxonomy} construct a comprehensive taxonomy of faults
in deep learning systems
through the analysis of 1059 artefacts from GitHub and Stack Overflow
and structured interviews with 20 practitioners.
Their five-category taxonomy (model, data, training, deployment, infrastructure)
covers the failure space at a coarser granularity than Wang et al.,
and several of their categories map onto the Wang crash types
that our VAL subcodes pre-empt.
Morovati et al.~\cite{morovati2023bugs} present \textsc{defect4ML},
a benchmark of 100 reproducible bugs
focused on TensorFlow and Keras.
The benchmark is constructed from GitHub and Stack Overflow reports
and serves as a fault-injection target for downstream tooling research.
Both taxonomies catalogue defects.
Our taxonomy catalogues the practices designed to catch them.

The closest empirical study of practitioner-authored validation in ML projects
is Openja et al.~\cite{openja_empirical_2024},
who investigate the test code present in eleven ML projects across four application domains.
Their study examines the formal test suites that ML projects ship
and reports that test coverage and structure
deviate substantially from conventional software projects.
Our study extends this finding in two directions.
First, we examine validation that is authored within notebooks
rather than in separate test files,
which is the dominant locus of ML development before production handoff.
Second, we include exploratory feedback alongside formal validation,
which captures a class of practice that test-suite analysis cannot observe
because exploratory feedback is structurally distinct from a unit test.
